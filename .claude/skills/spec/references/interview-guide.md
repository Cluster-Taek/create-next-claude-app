# 대화형 요구사항 인터뷰 가이드

`/spec`을 인자 없이 호출하면 대화형 인터뷰를 시작한다. AskUserQuestion 도구를 활용하여 Spec 작성에 필요한 정보를 수집한다.

## 인터뷰 흐름

### Step 1: 도메인 및 범위 파악

```
질문: "어떤 기능의 명세를 작성할까요?"
- 새로운 페이지/기능 (아직 구현 전)
- 기존 기능 변경/확장
- 기존 코드의 명세 역추출
```

후속:
- "도메인/페이지 이름은 무엇인가요?" (예: 사용자 목록, 사용자 등록)
- "산출물 저장 위치는 `docs/spec/` 가 맞나요?"

### Step 2: 페이지 개요

```
질문: "이 기능의 목적과 진입 경로는 무엇인가요?"
```

체크 항목:
- 목적 (한 문장)
- 진입 경로 (어떤 메뉴/버튼/링크에서 도달하는지)
- 특이사항/제약 (보안, 성능, UX 제약)

### Step 3: UI 영역 파악

```
질문: "이 페이지를 구성하는 UI 영역을 큰 단위로 나열해주세요."
```

영역 예시:
- Header / Footer
- 검색·필터 영역
- 목록/테이블 영역
- 입력 폼 영역
- 팝업/모달
- 패널 (이력, 코멘트, 정보 등)

각 영역마다 구성 요소(버튼, 입력 필드, 테이블 컬럼)를 수집한다.

### Step 4: 비즈니스 로직 수집

```
질문: "상태값, 분기 조건, 트리거가 있나요?"
```

체크 항목:
- 상태 정의 (예: 사용/미사용/재고/파기) — 각 상태의 의미와 발생 트리거
- 조건 분기 (IF/ELSE/CASE)
- 매핑 규칙 (입력값 → 출력값 변환)
- 외부 연동 (다른 도메인 상태 변화 시 연쇄 동작)

### Step 5: 페이지 이동/네비게이션

```
질문: "어떤 액션이 다른 페이지로 이동하나요?"
```

체크 항목:
- 행 클릭 / 버튼 클릭 → 어디로 이동
- 이동 시 상태 전달 (탭 활성화, 쿼리 파라미터 등)

### Step 6: Spec 검토

수집된 정보로 Spec 초안을 작성하여 사용자에게 보여준다. 사용자가 추가/수정하면 반영 후 파일로 저장한다.

저장 후 후속 단계 안내:
> TC 명세서를 작성하려면 `/testcase {파일경로}` 를 실행하세요.

## 질문 원칙

- 한 번에 1~2개 질문만 한다.
- 사용자가 모호하게 답하면 구체적인 예시를 제시하며 확인한다.
- 기술 구현이 아니라 **사용자 관점의 동작**과 **비즈니스 규칙**에 집중한다.
- 추측하지 않는다. 모르면 묻고, 사용자가 모르겠다고 하면 Spec에 `TBD`로 명시한다.

## Gherkin 입력 처리

사용자가 Gherkin 형식(Given-When-Then)으로 입력하면 다음으로 변환한다:

| Gherkin | Spec 위치 |
| :-- | :-- |
| **Given** | 사전 조건 → 비즈니스 로직 섹션 |
| **When** | 트리거 → UI 영역 액션 |
| **Then** | 결과 → 기대 동작 |
