# TC 명세서 템플릿

## 파일 헤더

```markdown
# 📋 {도메인} - {기능} 테스트 케이스 (TC)
```

예: `# 📋 사용자 관리 - 목록 페이지 테스트 케이스 (TC)`

## 표준 섹션 구조

영역별로 표를 분리한다. Spec의 영역 구조와 1:1로 매칭된다.

```markdown
## {N}. {영역명} ({English Label})

| ID                | 테스트 항목       | 테스트 시나리오                  | 기대 결과                      |
| :---------------- | :---------------- | :------------------------------- | :----------------------------- |
| **TC-{영역}-{NN}** | **{핵심 키워드}** | {사용자 동작/조건}              | {관찰 가능한 결과}             |
```

## ID 규칙

`TC-{영역코드}-{2자리 번호}` 형식

| 영역 예시          | 영역코드 | 예시 ID            |
| :----------------- | :------- | :----------------- |
| 목록 (List)        | LIST     | `TC-LIST-01`       |
| 등록 (Register)    | REG      | `TC-REG-01`        |
| 상세 (Detail)      | DET      | `TC-DET-01`        |
| 그리드 (Grid)      | GRID     | `TC-GRID-01`       |
| 팝업 (Popup)       | POP      | `TC-POP-01`        |
| 액션 (Action)      | ACT      | `TC-ACT-01`        |
| 패널 (Panel)       | PAN      | `TC-PAN-01`        |
| 제출 (Submission)  | SUB      | `TC-SUB-01`        |
| 검색 (Search)      | SRCH     | `TC-SRCH-01`       |
| 폼 (Form)          | FORM     | `TC-FORM-01`       |

새 영역이 등장하면 5자 이내의 약어를 만든다. 영역코드는 한 TC 파일 내에서 일관되게 사용한다.

서브 분기가 필요하면 `-1`, `-2` 접미사를 붙인다 (예: `TC-POP-07-1`).

## 컬럼 작성 규칙

### ID
- `**TC-XXX-NN**` 형식, 굵게.

### 테스트 항목
- 5~15자 이내의 핵심 키워드, 굵게.
- 예: `**검색어 미입력 시 노출 제한**`, `**권한별 메뉴 노출 제어**`

### 테스트 시나리오
- 사용자 행동 또는 시스템 조건을 한 문장으로 서술.
- 복수 단계는 번호 매김 (`1. ... 2. ...`)
- "~함", "~한 경우", "~시" 등 동작/조건형 어미 사용

### 기대 결과
- 관찰 가능한 결과를 한 문장으로 서술.
- 분기 결과는 줄바꿈(`<br>`)으로 구분
- 다중 검증 항목은 번호 매김

## TC 도출 규칙

Spec → TC 변환 시 다음을 따른다:

1. **각 UI 영역마다** 최소 1개의 표 섹션 생성
2. **각 분기/조건마다** 별도 TC로 분리 (IF/ELSE → 2개 TC)
3. **각 상태값마다** 렌더링/전이 TC 생성
4. **각 액션 버튼마다** 정상/비정상/조건부 TC 생성
5. **각 외부 연동마다** 트리거-결과 TC 생성

### 누락 검증

Spec의 모든 항목이 최소 1개 이상의 TC로 매핑되었는지 확인한다. 매핑되지 않은 요구사항이 있으면 사용자에게 보고한다.

## 하단 요약 섹션

```markdown
---

### 💡 주요 로직 및 체크 포인트

1. **{핵심 규칙 1}:** {요약 설명}
2. **{핵심 규칙 2}:** {요약 설명}
3. **{핵심 규칙 3}:** {요약 설명}
```

3~5개의 핵심 비즈니스 규칙을 요약한다. 테스터가 빠르게 컨텍스트를 잡을 수 있도록.

## 파일 명명 및 위치

- 파일명: `{도메인-소문자}-{기능-소문자}.tc.md`
  - 예: `user-list.tc.md`, `user-register.tc.md`
- 위치: 대응하는 Spec 파일과 동일 디렉토리 (`docs/spec/`)

## 체크리스트

TC 작성 후 다음을 확인:

- [ ] Spec의 모든 영역이 최소 1개 표 섹션으로 매핑됨
- [ ] 모든 분기/조건/상태가 개별 TC로 분리됨
- [ ] ID가 영역코드 단위로 순차 부여됨
- [ ] 각 TC가 단일 검증 포인트를 가짐
- [ ] 기대 결과가 관찰 가능한 형태로 서술됨 (모호한 "잘 동작" 금지)
- [ ] 하단 💡 요약에 핵심 비즈니스 규칙 3~5개가 정리됨
