# 대화형 인터뷰 가이드

`/tdd` 를 인자 없이 호출하면 대화형 인터뷰를 시작한다. AskUserQuestion 도구를 활용하여 요구사항을 수집한다.

## 인터뷰 흐름

### Step 1: 기능 파악

첫 질문으로 기능의 범위와 맥락을 파악한다.

```
질문: "어떤 기능의 테스트를 작성할까요?"
- 새로운 기능 (아직 구현 전)
- 기존 기능 변경/확장
- 버그 수정
```

후속 질문:

- "이 기능이 속하는 FSD 레이어와 슬라이스는 무엇인가요?" (예: features/auth, entities/user)
- "어떤 유형의 코드인가요?" (UI 컴포넌트 / 비즈니스 로직 / API 통합 / 유틸리티)

### Step 2: 정상 케이스 수집

핵심 동작을 명확히 한다.

```
질문: "이 기능이 정상적으로 동작하면 어떤 결과가 나와야 하나요?"
```

구체화 질문:

- "입력값은 무엇인가요?" (사용자 입력, API 응답, props 등)
- "기대하는 출력/동작은 무엇인가요?" (렌더링, 상태 변경, API 호출 등)
- "성공 후 다음 단계는 무엇인가요?" (리다이렉트, 알림, 상태 업데이트 등)

### Step 3: 엣지 케이스 수집

경계 상황을 탐색한다.

```
질문: "예외적인 입력이나 상황이 있나요?"
```

체크리스트로 확인:

- 빈 값 / null / undefined
- 최소값 / 최대값 / 경계값
- 중복 실행 / 빠른 연속 클릭
- 권한 없는 사용자
- 로딩 중 상태

### Step 4: 에러 케이스 수집

실패 시나리오를 파악한다.

```
질문: "실패하면 사용자에게 어떻게 알려야 하나요?"
```

체크리스트로 확인:

- 유효성 검증 실패 시 에러 메시지
- API 에러 (400, 401, 403, 404, 500)
- 네트워크 타임아웃
- 서버 미응답

### Step 5: 테스트 케이스 확인

수집된 정보를 기반으로 테스트 케이스 목록을 정리하여 사용자에게 보여준다.

```markdown
## 도출된 테스트 케이스

### 정상 케이스

- [ ] 올바른 이메일/비밀번호로 로그인 성공
- [ ] 로그인 성공 시 대시보드로 리다이렉트

### 엣지 케이스

- [ ] 빈 이메일 제출 시 필수 입력 에러
- [ ] 빈 비밀번호 제출 시 필수 입력 에러

### 에러 케이스

- [ ] 잘못된 자격증명 시 에러 메시지 표시
- [ ] 서버 에러 시 재시도 안내 표시
```

사용자가 목록을 확인하고 추가/수정하면 테스트 파일 생성으로 진행한다.

## 질문 원칙

- 한 번에 1~2개 질문만 한다. 질문 폭탄을 던지지 않는다.
- 사용자가 모호하게 답하면 구체적인 예시를 제시하며 확인한다.
- 기술적 구현이 아니라 사용자 관점의 동작에 집중한다.
- "~하면 어떻게 되어야 하나요?" 형태로 질문한다.
